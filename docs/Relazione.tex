\section{Introduzione}
Il progetto consiste nella realizzazione di un framework denominato \texttt{libmr}, scritto in linguaggio C, che implementa il paradigma MapReduce per l'analisi di file testuali su un singolo calcolatore. Il framework gestisce in modo autonomo la creazione di processi, la comunicazione tramite pipe e l'orchestrazione di thread C11, permettendo all'utente di concentrarsi esclusivamente sulla logica di \textit{mapping} e \textit{reducing}.

Il framework è ora completamente operativo e implementa tutte le funzionalità richieste, inclusa la pipeline multiprocesso e l'esecuzione multithreaded per la parallelizzazione del carico di lavoro.

\section{Architettura del Sistema}
L'architettura di \texttt{libmr} si basa su una pipeline di tre processi distinti che comunicano in modo unidirezionale tramite pipe POSIX:

\begin{itemize}
    \item \textbf{Processo Principale}: Agisce come coordinatore. Legge i file di input (singoli file o intere directory), invia le righe al processo Mapper tramite la pipe \texttt{main\_to\_mapper} e riceve i risultati finali dal processo Reducer tramite la pipe \texttt{reducer\_to\_main}.
    \item \textbf{Processo Mapper}: Riceve righe di testo dal Main. Un thread controller legge dalla pipe e inserisce le righe in una coda produttore-consumatore. Un pool di thread \textit{worker} (configurabile tramite \texttt{mr\_attr\_t}) preleva le righe, applica la funzione di \textit{mapping} fornita dall'utente e invia le coppie \texttt{<token, valore>} risultanti al processo Reducer tramite la pipe \texttt{mapper\_to\_reducer}.
    \item \textbf{Processo Reducer}: Riceve le coppie dal Mapper. Un thread controller raggruppa i valori per chiave utilizzando una tabella hash. Una volta terminato l'input, i gruppi di valori vengono inseriti in una coda per i thread \textit{worker} del reducer. Questi applicano la funzione di \textit{reducing} e inviano i risultati finali al processo Principale.
\end{itemize}

\section{Comunicazione e Sincronizzazione}
La comunicazione tra processi avviene tramite un protocollo binario semplificato implementato in \texttt{src/protocol.c}, che garantisce l'invio corretto di stringhe e metadati attraverso le pipe.

All'interno dei processi Mapper e Reducer, la sincronizzazione è gestita tramite:
\begin{itemize}
    \item \textbf{Code Bloccanti}: Implementate in \texttt{src/queue.c} utilizzando \texttt{mtx\_t} e \texttt{cnd\_t} (C11 threads) per gestire la distribuzione del lavoro tra thread controller e thread worker.
    \item \textbf{Mutex di Output}: L'accesso alle pipe di scrittura condivise tra più thread worker è protetto da mutex per evitare l'interleave dei messaggi.
\end{itemize}

\section{Sistema di Logging}
Il sistema di logging è sincronizzato e persistente. Le sue caratteristiche principali includono:
\begin{itemize}
    \item \textbf{Sincronizzazione}: Utilizzo di semafori POSIX (\texttt{sem\_t}) per garantire che le scritture sul file di log siano atomiche, anche quando provengono da processi diversi.
    \item \textbf{Formato}: Ogni riga di log segue il formato:
    \begin{center}
        \texttt{<timestamp> <pname>[<pid>] <tname> <level>: <message>}
    \end{center}
    \item \textbf{Configurabilità}: Il livello di log e il file di destinazione sono personalizzabili tramite gli attributi del framework.
\end{itemize}

\section{Test e Validazione}
La robustezza del framework è stata verificata attraverso una suite di test completa:
\begin{itemize}
    \item \textbf{Unit Tests}: Utilizzando \texttt{Unity}, sono stati testati i singoli moduli (code, protocollo, gestione attributi, logging).
    \item \textbf{Integration Tests}: Sono stati realizzati test di integrazione che simulano l'intero ciclo di vita di un job MapReduce, verificando la corretta terminazione dei processi e l'assenza di memory leak o deadlock.
    \item \textbf{Esempi Reali}: L'esempio \texttt{word\_count} incluso dimostra l'efficacia del framework su file di testo reali, producendo risultati coerenti con quelli ottenuti tramite strumenti standard (es. \texttt{wc}, \texttt{sort}).
\end{itemize}

\section{Conclusioni}
\texttt{libmr} rappresenta un'implementazione efficiente e modulare del paradigma MapReduce per sistemi POSIX. L'uso di processi separati garantisce isolamento, mentre l'impiego di thread all'interno di ciascun processo permette di sfruttare appieno le architetture multicore. Il framework si è dimostrato stabile e performante, rispettando tutti i requisiti di specifica iniziali.
